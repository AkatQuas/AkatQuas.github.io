Electron 应用的理想架构设计

May 9, 2022 · 11 min read

最近在折腾 Electron 相关的应用，一口气跟着 Udemy 的教程学了一些入门级的开发实践，如 Electron for Desktop Apps: The Complete Developer's Guide，或者 Electron From Scratch: Build Desktop Apps With JavaScript 。这些课程通过几个小巧的应用来介绍如何使用 Electron 开发，都是比较适合新手入门。

但是咱也想达到登堂入室的层次呀，VS Code 的代码实在难啃。所以找了几个项目复杂度中等的开源项目学习了一波。

Motrix​

Motrix 是一个界面非常友好的下载器，主要技术栈包括了 Electron 、 Vue 、webpack 等。

代码结构很清晰，解决问题时使用的 npm package 也很简单。值得学习。

下面简单说明一下 Motrix 中 Main 和 Renderer 的架构划分。

虽然 Main 里面没采用已有的 DI 框架，但是其中单例模式的使用处处可见。

扩展 Electron 类型的 d.ts 。

Main 负责的功能

  • 实例化 Launcher ，主要处理 app 层面的事件，如 文件打开，单例窗口 等。

  • 实例化 Application 应用模块，响应 renderer 的事件，然后交给各个单例去执行。

  • Application 继承于 Node 原生的 EventEmitter ，内部维护的单例包括有

    • 配置 Configuration （依赖于 electron-store）
    • 菜单 Menu
    • 系统托盘 Tray
    • 下载引擎 Aria2 （下载工具）
    • 程序坞 Dock（macOS 专属）
    • Touch Bar （macOS 专属）
    • 电池模块 （electron 提供的 powerSaveBlocker）
    • 文件处理 （应用打开 BT 文件直接打开时响应）
    • 窗口管理 （renderer 启动的 BrowserWindow 统一管理）

Renderer 负责的功能

主要负责用户的交互，主要使用了 Vue 作为 UI library。组件通过 Vuex 触发，再借由 ipcRenderer 与 Main 进行通信。

Vuex 承担了状态管理的职责，VueRouter 承担了 SPA 中页面切换的职责。

在 Main 中，ipcMain 监听到的事件又回通过 Application 进行 emit ，触发后续动作。

整体

套用主流 web 应用的视角，就能很容易理解 Motrix 的架构设计。

  • Main 相当于一个 Server，处理各种涉及 OS 的操作，如数据持久化、系统级的事件响应。
  • Renderer 相当于 Web 界面，响应绝大部分的用户交互，Renderer 几乎没有任何 OS 相关的工作量。
  • ipcMain / ipcRenderer 约等于网络，承载了通信职责。

Hyper​

hyper 的是一款基于 Electron 的命令行终端。

Renderer

可以说是标准的 React 应用了，技术栈主要是 react + redux。

终端实例 则封装了 XTerm。

Main

显然，Renderer 不太会执行很重的进程逻辑，它通过 ipcRenderer 向 Main 发送任务消息，Main 进程执行 exec 或者 execFile ，并将结果 stdout 和 stderr 的信息反馈回去。

通信 Message

作为一个终端，hyper 免不了要管理多个 BrowserWindow 实例。为了解决通信问题，hyper 在每个 BrowserWindow 创建的时候，main 进程一方面通过 Set 存储了该实例，另一方面给该实例增加了一个 rpc 属性 ，提供了 main->renderer 消息发送的快捷方式。 同时在 BrowserWindow 内部，window 全局也挂了一个 rpc 对象，提供 renderer->main 消息发送的快捷方式。

命令 commands

Main 和 Renderer 都维护了的命令集 Map<string,callback> 。这样一来，事件的调用就会变得很容易。

可能是因为没有具体场景，hyper 并没有实现注册机制。

Wexond Browser​

这是一个套 Electron 的浏览器。

Main

通过阅读 application.ts 的代码，不难发现其采用了单例模式。

StorageService 负责存储，具体媒介采用了 nedb 。

WindowsService 负责管理多个 AppWindow （封装的 BrowserWindow ） 实例，每个 AppWindow 实例通过 ViewManager 管理多个 BrowserView 实例，这些 View 就是最终呈现 浏览器地址 内容的页面载体。

DialogService 则会挂到 BrowserWindow 上，成为一些用户交互组件，例如 搜索框 。

Main 的 webpack.config.js 中有关于 Hot Reload Electron 的思路可以借鉴一波。

Renderer

AppWindow 会载入 app.html ，成为一个具有 Tab 栏 + 工具栏（含地址框） + 书签栏 + 页面载体的 初始化 窗口。app.html 其实占据的内容非常有限，只是窗口顶部一小部分。

app.html 加载完成后（跳过很多实例化过程），其内在 store service 会 ipcRenderer 发送一条 views-create-${windowId} 的事件，Main 进程中对应 windowId 的 BrowserWindow 会调用其专属的 ViewManager 创建对应的 tab ，创建完之后进行 select ，选择了激活的 tab view 页面 主体，同时会 setBounds 完全占据窗口的下半部分。

通信

项目使用了专门封装的 @wexond/rpc-electron 来解决通信问题，这样在 Renderer 和 Main 之间就不需要直接的 send / on 事件了，而且类型推导变得更加方便。思路是非常好的，具体实现可以看其源码。

VS Code​

VS Code 的 Renderer 比 Main 做的事情要重很多，也难怪 VS Code 逐渐开始吃内存了。

关于 VS Code，网上关于 VS Code 架构设计、源码分析的文章一抓一大把，而我鉴于能力有限，只能说一点点皮毛。

  • VS Code 自己实现了一套依赖注入了，DI 的使用极大地简化了代码之间的引用问题。
  • Renderer 进程不仅仅是个用户交互响应，还承担了大量的 OS 相关工作，举例如每个编辑器窗口都有自己的插件进程。
  • Event 的设计，使得事件监听变得容易溯源，不再会 on / emit 满天飞了，而且类型推导也变的很容易实现。
  • 命令系统 / Action 的设计，使得 service 的依赖可以在一定程度上解耦，也为后续 extensions 提供命令调用/命令注册的自由带来了可能性。（坏处就是 调用 不能在第一时间溯源到 注册）
  • ProxyChannel 取代了直接使用 ipcMain / ipcRenderer 的通信，能够支持 browser window 之间的通信。
  • （想到了再补充，其实就是鸽了的意思）

重新设计​

下面基于 Electron，设计一款我想要的桌面应用软件架构。

Main 和 Renderer 均采用某种 DI 框架。不同的 BrowserWindow 中，各自具有独有一颗 DI 树。（我就是这么喜欢依赖注入）

Main 必要地维护以下功能

  • Store 持久化数据的抽象
  • Message 消息中心，封装 ipcMain 的功能，承担 Main<-->Renderer 之间通信任务
  • Command 以命令系统代替 Service 之间的直接依赖
  • Context Main 执行的上下文，基本上就是 app 相关功能的扩展
  • Logger 日志系统
  • Network 统一的网络基础，方便代理的配置
  • WindowManager 管理各个 BrowserWindow 的实例
  • Configuration/Preference 配置内容
  • User 用户数据
  • Menu 封装 app 相关菜单功能，可以实现菜单的变更
  • AutoUpdater 应用自动更新机制

Main 可选的功能有

  • Metrics 监控系统
  • Theme 主题配色支持
  • Tray 封装 Electron 的 Tray 功能，实现系统托盘图标相关功能
  • Notification 封装后的通知功能，系统级别的消息通知
  • ChildProcess 某些任务通过子进程的形式去执行，减轻 Main 的压力
  • EventEmitter service 之间可能存在相互监听的需要，通过 生产者-消费者 的模式来解决问题

Renderer 则响应用户交互，形态为基础 web 应用 + Node features

  • 任意的前端框架、UI 库均可，呈现 web
  • Store 状态管理工具
  • Router 路由，页面动态切换
  • Network 必要的网络请求基础
  • Message 消息中心，与 Main 通信的唯一 “掮客” ，很多功能需要 Main 来执行或分发。

Renderer 可选的功能

  • ChildProcess BrowserWindow 也是个 Node 环境，因此可以将任务派给子进程。
  • Command 命令系统，解耦 Service，参考 vscode ，每个编辑器 window 都有一块自己的命令系统实例。
  • EventEmitter service 之间可能存在相互监听的需要，通过 生产者-消费者 的模式来解决问题

事实上，上述设计思路可以应用到 Electron 以外的任意桌面应用开发。
