重构 Electron 应用之后的感悟

March 25, 2022 · 4 min read

自从转岗以来，基于 Electron 开发编辑器应用已经快半年了，主要载体是 VS Code 。

现行的架构主要是利用 VS Code 提供的插件系统，注入相应的业务实现。该方案有很大的性能问题，是业务组件只能 VS Code 编辑器启动之后，才能初始化、实例化，性能有很大影响。另外，鉴于业务实现都跑在 VSC 的插件进程系统，通信显然会成为问题。

当业务复杂度上升之后，当前的设计方案会成为一个瓶颈。

这两个月刚好赶上了重构，主要思路就是将组件模块化， VSC 变成应用的一个模块，亦即编辑区，原先的插件式组件，升级成为与 VSC 同等级的模块。有这样的设计，应用的横向扩展性、模块间的联系性都能得到很大的提升。

那么我感悟到的是什么？并不是代码怎么重构、模块怎么实现，而是关于项目管理、展开、推进的。

Angular is the BEST! Nx.dev also!

  1. 对齐各方目标。

    要完成一个时间跨度长、工作量大的项目，第一条就是对齐各方目标，明确项目的目标，收益（向上管理的时候需要）。把各方的 KPI / 绩效考核 拉齐，这样容易让他们在这个事情上投入，也有理由给他们以反馈。

  2. 对齐项目方案。

    凡事欲则立，不预则废。项目方案没有考虑周全，各方协调没有到位，容易出现信息差，或者重复工作，最终就是导致返工，拖垮战线。

    如果是项目中的某一方，建议多投入，以全局的视角去分析，自身模块所处的位置，然后再确定方案，与整体方案相统一。

    如果是 Owner，则更要注意对齐各方的方案、信息了，因为整个项目的责任在自己身上。不然以后报告的时候，“负责 XX 项目，积极推动 YY” 就不好说咯。

  3. 小周期对齐进度，及时暴露风险点。

    这个算是老生常谈了，有的同事闷声不响的，一问才发现配环境配三天，开发进度 0 。

    在小公司经常搞每日站会就是这个道理，小地方耗不起的。（幺蛾子的另说）

差不多就这些了，结尾放个炒饭吧。无 P 无修，累了。
