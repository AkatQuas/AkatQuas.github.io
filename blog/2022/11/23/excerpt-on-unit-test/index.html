单元测试的实践

November 23, 2022 · 6 min read

概述​

之前 和 之前 对几个测试框架做过一些浅尝则止的练习，但是对于测试目的，测试方式不甚理解。最近读了一篇文章感觉颇有启发，结合一下其他学习资料，在此拾人牙慧。

软件工程中的测试，整体目标是为了保证项目代码的可用性，稳定性，功能符合预期。主要是通过自动化的手段，对代码、组件、应用进行测试，以确保代码的可用性与稳定性。

本文主要讲的是单元测试的实践。

测试思路​

从测试目的出发，反推测试过程，有利于测试用例的编写。

单元测试的测试对象通常是对项目代码中粒度较小的内容，常见的如函数，组件。

测试目的当然是该单元的运行与预期相符。

比方说：

  • 对于纯函数，给定输入，函数的运行结果应当和预期输出一致。

  • 对于组件，给定属性 props ，组件渲染结果中应当包含预期的元素。

事实上的某些待测试的函数会有副作用 side-effect，组件也会依赖其他组件，此时为这个单元进行测试的时候，就需要对它的依赖进行模拟，俗称打桩 stub (mock)。

对于不同形式的副作用，打桩形式也不一样。

副作用形式打桩思路
依赖并修改外部状态模拟状态，确保其在运行前后的变更符合预期。
依赖外部函数，但不依赖返回结果模拟函数，确保其调用参数、调用次数符合预期。
依赖外部函数，同时依赖对应返回结果模拟函数，并且以入参正确的模拟不同输出。确保待测函数按照返回结果的运行符合预期。
依赖子组件，仅渲染无交互shallow render，确保子组件存在即可。参考 enzyme shallow ， vitest stub and shallow
依赖子组件，依赖交互full render，子组件跟普通子元素地位相当。模拟触发事件功能，预期回调正确执行。参考 enzyme mount ， vitest component
依赖定时器 timer各个测试库均会提供 fakeTimer 。预期随着时间的变化，函数调用、状态变更应当符合预期。

测试实践​

纯函数​

纯函数的测试比较简单，不过编写测试用例时，需要把所有分支覆盖。

export function calculate(
  operand1: number,
  operand2: number,
  operator: '+' | '-'
): number {
  switch (operator) {
    case '+':
      return operand1 + operand2;
    case '-':
      return operand1 - operand2;
    default:
      throw new Error('Unknown operator');
  }
}

test('calculate', () => {
  expect(calculate(2, 3, '+')).toBe(5);
  expect(calculate(3, 2, '-')).toBe(1);
  // @ts-expect-error
  expect(() => calculate(2, 2, '*')).toThrow(/Unknown operator/);
});

外部状态​

为了便于测试函数运行时修改外部状态的情形，需要将状态导出，或至少提供一种方式能够读取状态。如果是其他全局状态同理。

// directly export
export const store = {
  count: 1,
};

// export an API for accessing
export const getCount = () => store.count;

export const increment = (value = 1) => {
  store.count += value;
};

// reset the store
beforeEach(() => {
  store.count = 10;
});

test('store', () => {
  expect(store.count).toBe(10);
  increment(20);
  expect(getCount()).toBe(30);
});

函数依赖​

对于依赖函数的场景可以有两种实践方式。方式一，观察。注意要将目标导出，并且在对象中观察。

export const howOld = (name: string, age: number) => {
  console.log(`${name} is ${age} years old`);
};

export const report = () => {
  howOld('Bob', 12);
};

// group as object so that jest can spyOn
import * as functions from './functions';
test('function', () => {
  const spy = jest.spyOn(functions, 'howOld');
  expect(spy).toHaveBeenCalledTimes(0);
  functions.report();
  expect(spy).toHaveBeenCalledTimes(1);
  expect(spy).toHaveBeenCalledWith('Bob', 12);
});

方式二，mock 函数。校验思路与前者一直。mock 函数时，也需要实现对应的返回结果，以达到测试目的。

export const report = (fn: (name: string, age: number) => string): string => {
  return fn('Alice', 12);
};

test('function', () => {
  const spy = jest.fn().mockReturnValue('default');
  expect(spy).toHaveBeenCalledTimes(0);
  expect(report(spy)).toBe('default');
  expect(spy).toHaveBeenCalledTimes(1);
  expect(spy).toHaveBeenCalledWith('Alice', 12);
});

子组件依赖​

子组件的依赖可以直接读文档 enzyme shallow 、 vitest stub and shallow 、 enzyme mount 、 vitest component 的例子，此处不再冗余。

测试核心目标主要包括：

  • 确保组件正确渲染；

  • 确保组件交互事件触发时，回调正确执行或状态变更符合预期；

  • 确保组件渲染时 hooks 等副作用正确执行。
