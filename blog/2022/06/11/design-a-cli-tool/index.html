设计一个 CLI 工具

June 11, 2022 · 7 min read

自从接手编辑器的 CLI 工具以来，我对原始的代码设计深恶痛绝。只是无奈重构、升级等迟迟无法提上日程，所以只能在这里做大梦。

常见工具​

在 Node 中，我接触过的 CLI 开发工具有

  • common-bin
  • commander
  • oclif

相比较而言，前两个 common-bin 和 commander 只能算是 CLI 的开发工具套件 (library) ，而 oclif 可以称得上 CLI 开发框架。

用 Web 生态去类比，前两者好比就是 React 和 Vue ，属于 Web UI library 。作为核心库，它们只提供了 MVVM 的功能，诸如 router 、store 、http 等都需要同生态下的其他 library 支持。

而 oclif 更接近 Angular 这种结结实实的 Web 框架。除了有开箱即用，省去绝大部分的工具类选型烦恼的优点，还有 schematics 这种好东西帮助开发提效。我这里有个 示例项目 可以带你入门 Angular 的 schematics。

设计 CLI​

抛开基础工具的选型，一款理想的 CLI 工具使用方式设计为

# 命令       全局选项         子命令          命令选项          参数...
command [global option] [sub command] [command option] [parameter...]

有一些文章会用 flag 来指代 option ，本文全用 option 。

执行过程 Execution​

在本设计中，CLI 的执行过程为

  1. 命令启动

  2. 载入相关配置，权重从 全局 - 项目 - 环境变量 - 选项 逐级升高

  3. 解析选项与参数

    1. 调整配置项，可选过程
    2. 确定具体执行命令
    3. 确定执行参数

  4. 执行命令，输出结果或处理异常

因此，要重点设计以下几个方面：

  • 配置 Configuration

  • 参数解析 Argument Vector

  • 命令执行 Command Execution

  • 错误处理 Error Handling

  • 交互 User Experience

配置 Configuration​

配置的作用主要是帮助确定当前命令执行的上下文环境 (Context) ，所以配置要尽可能早地完成实例化。

在后续的参数解析过程中，也许会对配置进行修改。这个修改依然要早于真实的命令执行。

前面提到配置的权重问题，其实就是一个配置覆盖问题。

全局配置

权重最低。全局配置主要是为了 兜底 (fallback) 或者 默认 (default) ，避免遇到执行时配置未定义的错误。

项目配置

权重高一级，可选。项目的存在能够实现项目之间的配置自定义，避免项目之间同时写全局配置而引起冲突。

环境变量

权重高一级，可选。环境变量的存在意味着在 CLI 真实执行时，临时修改某个配置项但是不改动配置文件。

选项

权重最高，可选。与环境变量的作用一样，只是呈现为不同的修改形式，修改时机。环境变量可以在配置实例化的时候就可以生效，而选项至少要等到参数解析之后，才能对配置项进行修改。

参数解析 Argument Vector​

参数解析主要作用是为了确定具体执行的命令接受到的是什么选项，什么参数。

解析主要就是校验存在性与合法性。

存在性是指传入的选项、参数是被定义过的，遇到无法识别的就抛出错误。

合法性是指选项、参数能够正确的转换到目标数据类型。

解析完成后会涉及到两个细节， 执行命令 和 命令参数 。

一级命令设计对绝大多数 CLI 来说已经足够了。复杂场景如 git 才涉及 子命令 (sub command) 。不过子命令的设计有助于划分命令的 模块 (module boundary) 。

命令参数的定位就等同于调用函数时的传参。这已经是属于具体逻辑的范围了，不做展开。

命令执行 Command Execution​

命令执行可以是由一级命令直接调用具体函数来完成。这种做法显得太朴素了。

借助策略模式，命令执行通过分发，在具体实现类中去执行。分发的形式可以与前面提到的 模块划分 结合起来，有利于代码结构设计。

错误处理 Error Handling​

在 执行过程 中可以看到整体过程是串行的。因此错误处理设计显得比较容易：

  • 整体上做一个全局兜底错误处理。

  • 流程的每一步都可以抛出各自的错误处理。

严格来说，错误处理不算是 CLI 工具必要内容，但是良好的错误处理设计有益于对 CLI 的整体架构。

交互 User Experience​

部分场景下，需要与用户进行交互或者信息展示，如询问密码 (prompt) ，确认步骤执行 (confirm) ，进度条 (progress) 等。

良好的交互反馈能够降低用户执行命令时的等待焦虑。

其他 Miscellany​

以下省略一些比较小的细节设计，有兴趣的读者建议自己实践、自己总结。

  • 命令别名 (command alias)
  • 测试 (testing)
  • 调试 (debugger)
  • 日志 (logger)
  • 文档 (documentation)
  • 输出格式 (output)
  • 自动补全 (<tab> autocomplete)

总结 Wrap up​

从软件的层面来看，CLI 工具只是一个很小的应用。那些适用于设计一个健壮的应用的原则，同样适用于 CLI 设计。
