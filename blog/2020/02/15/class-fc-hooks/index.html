Class Component ?= Function Component &plus; Hooks

Main Content‚Äã

Long time ago, I came across to this awesome article about several architectures used in front end from the beginning of PC.

In summary, what the front end mainly concerns are the following things:

  1. The current data, also known as state,

  2. The current view, which is user interface,

  3. Reaction to the user action, such as mutating data, animating the view, even side effects,

  4. Rendering the view according to the current data, in another phrase, synchronizing the view to the data.

In React, we mainly write two kinds of component: stateful Class component and stateless Function component. The differences between these two are not only the state maintained by the component, but also the lifecycle events. So what are the lifecycle events?

Before talking about the lifecycle events, we need to understand some basic concepts in React.

When it comes to rendering, there are three concepts we need to keep in mind, Component, Elements, Instances. here is the official explanation on their differences.

An element is a plain object describing what you want to appear on the screen in terms of the DOM nodes or other components. Elements can contain other elements in their props. Creating a React element is cheap. Once an element is created, it is never mutated.

So we have to re-create a new element, (indeed, every time when the render function is invoked) if we need to update the DOM nodes. The React helps us to differ elements and patch the differences to the DOM nodes.

A component can be declared in different ways. It can be a class with a render() method. Alternatively, in simple cases, it can be defined as a function. In either case, it takes props as an input, and returns an element tree as the output.

A component tells React how to create the elements using props and states, if necessary. But we cannot use the component directly, we need to instantize it.

An instance is what you refer to as this in the component class you write. It is useful for storing local state, (local property/method) and reacting to the lifecycle events.

Function components don't have instances at all.

Class components have instances, but you never need to create a component instance directly‚ÄîReact takes care of this.

Why we have to instantize the Class component?

The function components don't need instances because all the data, which used to create elements, can be accessed directly from the function invocation arguments. It means we don't need to create closure to preserve the references.

However, we need the references to the local state, local property/method when using Class components. We have to instantize an instance of a Class component. The render function is one of the local property/method belongs to this instance. So it can access to this instance easily.

class MyComponent extends Component {
  constructor(props, context) {
    super(props, context);
    this.state = {};
    this.nonStateProperty = 'non state property';
    this.methodA = this.methodA.bind(this);
  }
  methodA() {}

  render() {}
}

The above codes show how we to bind the local method to the specific instance. The bind statement is necessary in React for that JavaScript has the magic this. It's implicit the instance is encapsulated as this during the runtime. Believe it or not, it is a special case of closure.

Besides, local state, the Class components have the lifecycle events.

lifecycle events are those you can override to run code at particular times in the process.

Summary from the official document indicates that when something changes, the lifecycle events would be triggered.

Here is list of the types of change and the according lifecycle method, which would help you to understand when the certain lifecycle events will be invoked:

  • element patched to DOM nodes:

    • componentDidMount, which is invoked immediately after a component is mounted (inserted into the tree), aka, null -> new node.

    • componentDidUpdate, which is invoked immediately after updating occurs. This method is not called for the initial render, aka, old node -> new node.

    • componentWillUnmount, which is invoked immediately before a component is unmounted and destroyed, aka, old node -> null.

  • props/state updates:

    • static getDerivedStateFromProps, synchronized function which should return an object to update the state, or null to update nothing.

    • deprecated componentWillReceiveProps, Use this as an opportunity to perform preparation before an update occurs. This method is not called for the initial render.

    • componentDidUpdate, after the render function called to create the possible new elements, it will be invoked after updating occurs.

By using the name lifecycle, we could imagine that the instances of Class components has a lifetime to span. But, how about Function components, what do they have if they don't have lifecycle events?

Actually, Function components have lifecycle as well. To understand this saying, we have to rethink what the lifecycle really is.

Lifecycle methods are the response action to certain changes and events.

In simple words, when the state/props changes, we have to re-create elements. But, we could make some actions before, during and after the re-rendering process.

const FunctionComponent = (props) => {
  // doing side effects
  return <ComponentTree />;
};

Before hooks come out, we can only make the actions before the re-rendering process. The equivalent lifecycle method is componentWillReceiveProps. There is not much we can do after the re-rendering process.

Here comes the useEffect to rescue. This article, written by Dan, A complete guide to useEffect, shows how useEffect works, specifically, staring from here;

function Counter() {
  const [count, setCount] = useState(0);

  const increment = useCallback(() => {
    setCount((c) => c + 1);
  }, []);

  useEffect(() => {
    document.title = `You clicked ${count} times`;
  });

  return (
    <div>
      <p>You clicked {count} times</p>
      <button onClick={increment}>Click me</button>
    </div>
  );
}

  1. React: Give me the UI when the state is 0.

  2. The component:

    Here‚Äôs the render result: <p>You clicked 0 times</p>. Also remember to run this effect after you‚Äôre done: () => { document.title = 'You clicked 0 times' }.

  3. React: Sure. Updating the UI. Hey browser, I‚Äôm adding some stuff to the DOM.

  4. Browser: Cool, I painted it to the screen.

  5. React: OK, now I‚Äôm going to run the effect you gave me.

    Running () => { document.title = 'You clicked 0 times' }.

Now we click the button:

  1. The component: Hey React, set my state to 1.

  2. React: Give me the UI for when the state is 1.

  3. The component:

Here‚Äôs the render result: <p>You clicked 1 times</p>. Also remember to run this effect after you‚Äôre done: () => { document.title = 'You clicked 1 times' }.

  1. React: Sure. Updating the UI. Hey browser, I‚Äôve changed the DOM.

  2. Browser: Cool, I painted your changes to the screen.

  3. React: OK, now I‚Äôll run the effect that belongs to the render I just did.

    Running () => { document.title = 'You clicked 1 times' }.

The function defined in the useEffect is different on every render, because no dependency is specified.

More generally, effects could:

  1. run after every render, because it's specified by the rendering mechanism

  2. are conceptually a part of the component output, because it can do some side effects after rendering element.

  3. ‚Äúsee‚Äù the props and state from that particular render, because it creates a closure when creating the effect function.

React synchronizes the DOM according to our current props and state, it just creates/re-creates elements according to our jsx code. There is no distinction between a "mount" or an "update" when rendering.

Whilst, useEffect lets you synchronize things outside of the React tree according to our props and state. When looking back at, take componentWillReceiveProps, componentDidUpdate as examples, we used to do these synchronizations in these lifecycle events.

So useEffect is not something that replicate the lifecycle events such as componentDidMount, and their mental model is closer to implementing synchronization than to responding to lifecycle events. Trying to find effects exact equivalents with lifecycle events may confuse you more than help.

In Class component, we can access to the state by referring to this in a instance, implicit closure.

By using useState in Function components, we also create a closure to encapsulate the state. Also, React ensures the setState function stable for the render, still with the help of closure. (So does the dispatch generated by useReducer as long as the reducer stable.)

The instance methods are bound to this instance. Whilst hooks provide us with useCallback, which also would create closures to encapsulate the props/state it can access to in the scope.

Conceptually, the instances of the Class components implicitly create closures for us to keep the reference to the methods, props, state. Whilst, hooks have to do the closure encapsulating since there's no instance in Function components.

In the end‚Äã

Is Class component the combination of Function component and hooks?

The answer is no. They both could work very well with their own pros and cons. They are just two means of tackling the above concerns.

It's fine to use either of them, as long as it meets your requirements.
