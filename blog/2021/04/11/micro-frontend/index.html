Micro Frontends

April 11, 2021 · 6 min read

Micro frontends deliver the same flexibility and maintainability to browser-based applications that microservices provide for backend systems.

The whole project is viewed as a set of standalone components, built upon any tech-stack or framework, with their own interfaces, logic, and storage. Then these mini-applications are evolving independently and will be composed together shipping to the browser.

There are several tech stacks developers can adopt to build micro frontends architecture.

Client-side integration​

Separate pages​

It's also known as Multi-Page Application, MPA.

This is simplest and easy-implementation architecture. Every application is served as a standalone service, which produce complete server-rendered HTML documents. They are connected via <a> tags.

Hard navigation among pages are triggered by clicking a link, reloading the complete page then displaying the desired content.

<!-- in app 1 -->
<div id="main-app1">
  <div id="content">
    <!-- content provided by main-app -->
    <a href="http://link-to-app2"></a>
  </div>
</div>

<!-- in app 2 -->
<div id="main-app2">
  <div id="content">
    <!-- content provided by main-app -->
    <a href="http://link-to-app1"></a>
  </div>
</div>

Iframes​

Iframes allow embedding one page into another page while maintaining the same loose coupling and robustness properties that the link integration provides.

Iframes come with strong isolation, which means what happens in the iframe stays in the iframe. But they also have significant drawbacks, such as layout constraints, performance overhead, communication complexity, SEO confusion.

<div id="main-app">
  <div id="content">
    <!-- content provided by main-app -->
  </div>
  <div id="ads">
    <iframe src="http://ads.com/{some-ads}"></iframe>
  </div>
</div>

Ajax​

Client-side JavaScript triggers the Ajax calls to fetch server-side generated HTML snippets, and then update document dynamically and partially.

Be careful, HTML5 specifies that a <script> tag inserted with innerHTML should not execute.

Using Web Components together with Ajax for internal updating is also a good fit.

Sample code.

// main app
window
  .fetch('/some-url')
  .then((res) => res.text())
  .then((html) => {
    document.querySelector('to-update').innerHTML = html;
  });

Web Components​

Web Components is an umbrella term. It describes three distinct new APIs: Custom Elements, Shadow DOM, and HTML Templates.

Web Components provides the ability to encapsulate style and business logic.

Sample code, Sample code2.

<div id="main-app">
  <custom-button type="primary"></custom-button>
</div>

<script src="http://js-define-custom-button"></script>

<!-- define the custom button -->
<script>
  window.define(
    'custom-button',
    class extends HTMLElement {
      static get observedAttributes() {
        return ['type'];
      }

      constructor() {
        super();
      }

      adoptedCallback() {
        console.log('Custom button moved to new page.');
        // using shadow DOM if necessary
      }

      disconnectedCallback() {
        console.log('Custom button removed from page.');
      }
    }
  );
</script>

Unified SPA​

The Unified SPA describes a single-page application composed of other single-page applications.

It requires all the components to be a single-page app, using Sveltejs, Angular, React, Vue, etc.

A parent application, which is often called the app shell, will unify these single-page apps into one monolithic application.

The app shell includes the single-spa JavaScript code and maps URL prefixes to the code of a specific application.

Single SPA, Ara framework.

<html lang="en">
  <head>
    <title>SPA demo</title>
    <script src="/single-spa-base.js"></script>
  </head>
  <body>
    <div id="app1-holder"></div>
    <div id="app2-holder"></div>
    <div id="app3-holder"></div>

    <script type="module">
      singleSpa.registerApplication(
        'app1',
        () => import('http://app1/pages.min.js'),
        ({ pathname }) => pathname === '/'
      );
      singleSpa.registerApplication(
        'app2',
        () => import('http://app2/pages.min.js'),
        ({ pathname }) => pathname.startsWith('/app2/')
      );
      singleSpa.registerApplication(
        'app3',
        () => import('http://app3/pages.min.js'),
        ({ pathname }) => pathname.startsWith('/app3/')
      );
      singleSpa.start();
    </script>
  </body>
</html>

Server Side Integration​

Routing with proxy​

Nginx acts as a proxy and forwards the requests to the responsible services. This could be used to unify MPA architecture.

http {
  upstream app1 {
    server localhost:3001;
  }

  upstream app2 {
    server localhost:3002;
  }

  log_format compact ':3000$uri $status';

  server {
    listen 3000;

    # comment out on windows
    access_log /dev/stdout compact;

    location /app1/ {
      proxy_pass  http://app1;
    }

    location /app2/ {
      proxy_pass  http://app2;
    }

    location /app2-new {
      proxy_pass  http://app2;
    }
  }
}

SSI, ESI, Podium​

This is similar to client-side integration Ajax.

http {
  upstream app1 {
    server localhost:3001;
  }

  upstream app2 {
    server localhost:3002;
  }

  log_format compact ':3000$uri $status';

  server {
    listen 3000;
    ssi on;
    ; ...
  }
}

      <!--#include virtual="/app2/fragments/ads" -->
    </aside>
  </body>
</html>

<!-- app2 html fragment for ads -->
<a href="http://ad-detail">
  <div>ads</div>
</a>

Universal rendering​

Universal composition is the combination of server-side and client-side composition technique.

For the first request, a technique like SSI, ESI, or Podium assembles the markup of all micro frontends server-side. The complete HTML document gets sent to the browser. In the browser, each micro frontend hydrates itself and becomes interactive. From there on, all user interactions can happen fully client-side. The micro frontends update the markup directly in the browser.

Angular universal, Next.js.

Sample code: SSI in server-side, ajax and web-component in client side.

Development​

Only the app-shell server is running, all the other app-parcels would generate their static assets (js bundles, images, style assets) into app-shell's output directory.

Distribution​

Build app-shell first, then every app-parcel.

Concerns​

  • Global state.
  • Style isolation: BEM, css module, shadow DOM.
  • Communication among app-parcels: event bus, broadcast channel, event bus.
  • Shared vendor or dependencies: unified tech stack, vendor version.
  • Assets updating: cache invalidation strategy.

Glossary​

namedescription
app-shellapp shell for layout, responsible for micro apps register/unregister
app-parcelmicro autonomous interactive applications

My Opinion​

Micro frontends majorly focus on the integration. Application integrates components. Micro-frontend integrates micro-apps.

It's recommended to do the integration during building/distributing stage, so the end user feel little latency when using the application.
