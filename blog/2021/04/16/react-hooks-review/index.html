React Hooks Review

April 16, 2021 · 5 min read

Recently, I get a little confused on how React hooks executes, and this is what I have summarized.

During the process of delving, I come across some knowledge on how V8 deals with function compilation, since we would have lots of anonymous functions (closures in some cases) defined whilst using hooks.

Get started​

Imagine we have a class component like the following one.

class Counter extends React.Component {
  state = {
    counter: 1,
  };
  render() {
    const { counter } = this.state;
    return (
      <>
        <p>current value: {counter}</p>
        {/* anonymous function */}
        <button onClick={() => this.setState({ counter: counter + 1 })}>
          Increment
        </button>
      </>
    );
  }
}

It's obvious that button would receive a new click callback after every render.

To avoid repeatedly generating the onClick anonymous function, so we have another style of function definition.

In V8, the anonymous function is only parsed once, but allocated every time execution hits the code. So we have different function every time the code runs.

class Counter extends React.Component {
  state = {
    counter: 1,
  };
  // class method
  increment = () => {
    const { counter } = this.state;
    this.setState({
      counter: counter + 1,
    });
  };
  render() {
    const { counter } = this.state;
    return (
      <>
        <p>current value: {counter}</p>
        {/* reference to the class method */}
        <button onClick={this.increment}>Increment</button>
      </>
    );
  }
}

In Functional Components, we have similar situation.

const Counter = () => {
  const [counter, setCounter] = React.useState(1);
  return (
    <>
      <p>current value: {counter}</p>
      {/* anonymous function */}
      <button onClick={() => setCounter(counter)}>Increment</button>
    </>
  );
};

Still, button would receive a new click callback after every render.

Here comes the useCallback to rescue.

const Counter = () => {
  const [counter, setCounter] = React.useState(1);
  const onClick = React.useCallback(() => {
    setCounter(counter + 1);
  }, [counter]);
  return (
    <>
      <p>current value: {counter}</p>
      <button onClick={onClick}>Increment</button>
    </>
  );
};

useCallback would return the same function as long as the dependency counter doesn't change.

So the VDOM reference stays untouched even if the function component gets re-rendered.

Now, we are going to look at a more complicated situation.

const Counter = () => {
  const [counter, setCounter] = React.useState(1);
  const [tip, setTip] = React.useState('first tip');

  const onClick = React.useCallback(() => {
    setCounter(counter + 1);
  }, [counter]);

  React.useEffect(() => {
    setTip('second tip');
  }, []);

  const [heavy, setHeavy] = React.useState({
    a: {},
    // ... some huge object
    z: {},
  });

  return (
    <>
      <p>current tip: {tip}</p>
      <p>current value: {counter}</p>
      <button onClick={onClick}>Increment</button>
    </>
  );
};

It's guaranteed there would be two render invocations. Here come the questions：

Question 1. Will the onClick functions equal to each other during these two invocations?

My Opinion

They are same functions. useCallback would return the same function as long as the dependency counter doesn't change. Internally, hook.memoizedState is not updated, so the same function is returned from same memoizedState.

Question 2. Will React.useCallback execute twice? Will each execution receive same arguments (anonymous function)?

My Opinion

useCallback is just a plain normal JavaScript function, so it executes twice.

However, the arguments are different. There will be a newly allocated local function in every execution.

Question 3. Will React.useEffect execute twice? Will each execution receive same arguments (anonymous function)?

My Opinion

useEffect is just a plain normal JavaScript function, so it executes twice.

And, each execution is invoked with a brand new allocated local function. Internally, useEffect would adjust the effect flag in hook.memoizedState according to the dependencies.

Question 4. Will React.useState execute twice? Will each execution receive same arguments (a large object)?

My Opinion

useState is just a plain normal JavaScript function, so it executes twice.

Internally, useState would memorize the value in the first execution. The arguments is ignored in the rest executions.

useState would receive a new allocated large object as arguments on each execution. And this object is eligible to be collected (Garbage collection) after execution.

Question 5. What's point of using these useCallback, useMemo, useEffect hooks? What we could do if the arguments are different?

My Opinion

hooks invocation is required by React fiber reconciliation.

Without these hooks invocation, React is unable to trace these local variables/closure objects. So React cannot perform reconciliation actively upon these objects' mutations.

With the help of these tracing, React fiber keeps its references solid, avoiding unnecessary reconciliation.

The function component might re-render and return a new VDOM. However, there will be no difference during VDOM diff comparison sine their references are same. This would result no DOM update execution, which is skipped.

The following example is more persuasive：

const Child = ({ onClick }) => <button onClick={onClick}>Click me!</button>;

const ParentWithoutHooks = () => {
  const handleClick = function () {
    console.log('clicked');
  };
  // whenever parent re-renders,
  // the Child is forced to rerender
  // since handleClick is a new function after every render
  return (
    <>
      <Child onClick={handleClick} />
    </>
  );
};

const ParentWithHooks = () => {
  const handleClick = React.useCallback(
    function () {
      console.log('clicked');
    },
    [
      /* dependencies if necessary */
    ]
  );
  // when the parent re-renders,
  // in an optimistic situation,
  // Child is only re-rendered if onClick is changed
  return (
    <>
      <Child onClick={handleClick} />
    </>
  );
};

Summary​

Most of the time it does not matter. Pay more attention to the business.
